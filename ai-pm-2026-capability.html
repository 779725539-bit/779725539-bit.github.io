范式转移

PRD → 原型 + 标准 + 评测

能力一

🛠️ 原型 — 从 PRD 到 Vibe Coding

产品经理自己写代码？

不是写代码，是用 AI 把想法变成可交互的原型

  • 利用 Dify / LangChain / Cursor 等工具快速构建 MVP
  • 跨越"PM画原型→设计师出稿→开发写码"的漫长链路
  • 原型代码可直接交给 Coding Agent 参考，提升协作效率
  • 从"写文档说服别人"变成"做出来让人体验"
关键转变：传统PM的武器是Word，AI PM的武器是一个跑得起来的Demo。
不是效率工具，是与产研团队配合的基本功。

能力二

📐 标准 — 定义"好"的维度

没有标准，就没有方向

Benchmark = 考试大纲，告诉团队"我们追求什么"

  • 根据产品类型定义评价维度：格式规范性 · 信息完整性 · 准确性 · 可读性 · 安全性
  • 不同场景标准不同：高管报告追求"精简精炼"，调研报告追求"全面广度"
  • 不同用户标准不同：小白需要"手把手引导"，专家需要"极简高密度"
  • Benchmark 是产品战略的起点，直接决定迭代方向
核心洞察：AI 产品最大的坑不是"做不出来"，是"不知道什么算好"。
先定标准，再动手——这是区分 junior 和 senior AI PM 的分水岭。

能力三 · 最核心

🧪 评测 — AI PM 的核心交付物

Eval 取代 PRD

一份好的 Eval 同时扮演三个角色

  • 📋 产品规格 — 定义功能边界和质量标准
  • ✅ 验收标准 — 可自动化执行，不依赖人工判断
  • 🗺️ 产品路线图 — Eval 没过的case = 下一步要做的事
实例："从视频生成食谱"功能的 Eval →
· 食材是否全部列出？（字符串匹配）
· 步骤是否简洁？（AI 评审员打分）
· 过敏原是否标注？（规则校验）
每一条都是可执行、可重复、可自动化的测试信号。

核心工作流

Observe → Analyze → Evaluate → Improve 飞轮

1

👀 观察 Observe

监控线上数据，收集用户输入、
模型输出及失败案例（Bad Case）

2

🔍 分析 Analyze

识别问题模式，判断是 Prompt、
模型还是流程导致的问题

3

🧪 评测 Evaluate

将问题转化为新的 Eval 测试用例，
必要时调整 Benchmark 标准

4

⚡ 改进 Improve

优化 Prompt / 微调模型 / 改进工作流，
上线后重新观察，飞轮加速

更多数据 → 更好的评测 → 更精准的改进 → 产品持续进化 🔄
