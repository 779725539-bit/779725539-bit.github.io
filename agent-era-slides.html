核心判断

不是卷 Chat，而是卷 Agent。

Chat 时代

比的是谁更会回答

单位 = 单轮回答

指标 = 回答质量、表达、体感

护城河 = 模型本身

Agent 时代

比的是谁更会执行

单位 = 整条任务链路

指标 = 完成率、稳定性、返工率、总成本

护城河 = 模型 + 框架 + tools + memory + 生态

真正正确的事，不是做一个"更复杂一点的 Chat"，而是在复杂 Agent 框架上端到端完成高复杂度任务。 — 罗福莉 · 03:04:09

关键判断 #2

Agent 框架是分水岭，不是附属品。

框架不是 UI 壳，不是产品皮，不是单个 prompt。
它是人和模型之间的厚中间层。

交互层

定义你怎么和 Agent 对话

编排层

定义 Agent 怎么跟模型沟通、调度工具、管理记忆

← 这一层可以做得非常厚重

展示层

前端 UI 反而是最薄的一层

我自己其实会把 OpenClaw 当作一个划时代的 Agent 框架。真正发生一个转变，是我去用它那一刻。 — 罗福莉 · 02:25 / 03:28
4 / 10

深层洞察

框架在补模型的「行动缺陷」。

模型会说，但不一定会做。

框架通过编排、记忆、调度和执行规范，
把模型从「会回答」提升到「会执行」。

✦ 任务编排 → 拆解复杂链路
✦ 工具调度 → 调用外部能力
✦ 记忆管理 → 跨回合状态保持
✦ 执行规范 → 提升复杂任务正确率
我认为 OpenClaw 框架的设计之初，它其实是想尽量地通过 Agent 的整套编排，来弥补模型的短板。 — 罗福莉 · 298
它是补齐了模型在复杂任务上完成的正确率。 — 罗福莉 · 2490

更准确的说法：
框架在把模型能力组织成可执行能力。

Long-context

Long-context 的价值不在「更长」。

大众理解 ❌

• 上下文越长越强

• 能塞更多文档就更厉害

• 窗口够大，复杂任务自然变好

罗福莉判断 ✅

• 真正超长上下文任务大多是复杂任务

• 关键不是塞满，而是有效利用

• 要低幻觉理解 + 持续状态 + 推理效率

• 成本够低，速度够快

未来最稀缺的能力，不是更长的 context，
而是可负担、可持续、可引用的长任务理解能力。

6 / 10

训练范式 + 系统接口

Post-train 变重，Skills & Memory 是关键。

Post-train

不只是对齐，更像：

行动能力训练 环境适应 执行规范形成 复杂任务稳定性

这些执行规范通常不在互联网预训练数据里

Skills

人和 Agent 共创的关键接口

隐性经验 → 外显化 人的经验 → 可调用规范 互联网外的另类信息

人和 Agent 共同沉淀工作方法

Memory

让 Agent 越用越聪明

分层持久记忆 人物画像 任务状态沉淀 跨窗口上下文

Memory 做得好 = 模型能力强的表现

7 / 10

竞争格局

1T 是入场券，不是终局。

基座模型

≥ 1T + 强 code 能力
= 站上同一条竞争线

Post-train

Agent RL + 执行能力
= 让模型会「做」

Agent 框架

编排 + 工具 + 记忆
= 组织执行能力

生态整合

模型 + 框架 + tools + memory
= 真正的护城河

只要具备大于 1T、且 code 能力很强的基座模型，大家就基本站上了同一条竞争线。 — 罗福莉 · 56:21

基座不能错，但胜负不再只由基座决定。

组织与用人

环境比经验更重要。

她的三个组织判断

1. 固定分工会扼杀创造力

不该按 Pre-train / Post-train 硬切
组织方式会被范式重塑

2. 最有效的管理是热爱驱动

不是层层汇报，是找到热爱这件事的人
然后给他们空间

3. 环境比经验更重要

很多能力 1-4 个月就能快速习得
决定成长的是环境，不是资历

这些能力最多一两个月，慢的话三四个月，确实都可以被快速习得。所以环境反而比经验更重要。 — 罗福莉 · 01:15
最有效的管理方式是热爱驱动。 — 罗福莉 · 02:01:26

对我们的启示：

不招有经验的，招有热爱的。
